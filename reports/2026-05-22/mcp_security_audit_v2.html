1. Resumen ejecutivo refinado

12 MCP servers conectados al daemon vía bridge OAuth de claude.ai. El paper NSA confirma cuatro deltas críticos que tu stack ya rozaba pero ninguno tiene un control formal hoy:

  1. Output filtering en pipelines encadenados — el paper es explícito: "Each output must be treated as untrusted input to the next phase of the pipeline". Tu wrapper untrusted_context ya existe para wiki/learning_loop pero no se aplica a output de MCP tools. Cierra el vector dominante (semantic poisoning + cascading prompt injection).
  2. Approval workflow para cambios de capability — "a change in capability or data access for an MCP server that is already trusted or connected often can be made without approval". No tienes detección de cambio en el set de tools expuesto por un server.
  3. Audit logs MCP-específicos — el daemon tiene observe_stream + task_ledger pero no consolida un audit log per-MCP-server con parámetros, identidad, y hash del output (NSA: "exact parameters, identities involved, and where feasible cryptographic hashes of results or output").
  4. Filtering outgoing proxy / DLP — la guía dice usar Squid/tinyproxy/DLP enterprise para connections salientes del entorno MCP. No hay nada hoy; los MCP servers de claude.ai usan TLS y endpoints arbitrarios.
Vector real más cercano a tu stack: "Poisoning output for downstream automation" (NSA p. 9) + WhatsApp MCP exfil pattern (NSA p. 9). Un actor que ponga contenido en un Figma file, un comentario de Linear, una página Notion compartida, o un email entrante puede meter instrucciones que el brain procese como user input. Tu mitigación actual (defensive prompt + Tier 3 gates) NO cubre este vector porque opera después de que el contenido entró al contexto.

3. Mapeo NSA → daemon · 8 security concerns

3.1 Access control (NSA p. 4)

"Associating a session to an identity is not defined by the protocol... Many implementations omit authentication entirely, and those that do include it often lack any role-based enforcement (CRUD permissions)."

Tu stack: OAuth via claude.ai cubre auth de cuenta. RBAC granular per-tool no existe — Linear puede leer issues de cualquier team al que tengas acceso, Notion puede fetch de cualquier página del workspace. Acción: los workspace filters de los gates Tier 3 deben checar destination scope (team_id, database_id, project_id) y bloquear writes fuera de scope Pachano.

3.2 Insecure context / data serialization (NSA p. 4-5)

"Serialized content including comments or prompts may open a path for injection techniques."

Tu stack: los outputs MCP llegan al brain como tokens. Acción: wrapper untrusted_context source="mcp_tool_result" en TODO read tool (sección 5.2).

3.3 Poor approval workflows (NSA p. 5)

"A change in capability or data access for an MCP server that is already trusted or connected often can be made without approval."

Tu stack: hoy el daemon NO detecta si un MCP server agregó/cambió tools entre conexiones (ej: si Vercel suma un tool nuevo de billing, no te enteras). Acción: hash el set de tools por server al instalar, alertar Tier 3 si cambia.

3.4 Token / session security (NSA p. 5-6)

Tu stack: claude.ai maneja OAuth bearer tokens; rotación + revoke vive en su panel, no en el daemon. Acción: documentar revoke path por server en ~/.claw/mcp-receipts/.

3.5 Misconfigurations / poor implementation (NSA p. 6)

Tu stack: trampa principal = los 3 servers dormidos (Daloopa, HubSpot, Windsor). Si completas OAuth sin redefinir gates, repites el patrón. Acción: bloqueo preventivo — definir gates antes de completar auth.

3.6 Inconsistent behaviors (NSA p. 6-7)

"This divergence, driven by probabilistic interpretation of prior context, can be exploited by a malicious actor who preconditions the agent to arrive at a specific or unsafe outcome."

Tu stack: riesgo medio. El brain procesa contexto extenso (session_state, reply_context, memory). Acción: reforzar la regla actual de "memoria/learning_loop ≠ instrucciones" extendiéndola explícitamente al output de MCP tools.

3.7 Poor or missing audit logs (NSA p. 7)

"Logs should capture a traceable sequence of actions across sessions... cryptographic hashes of results or output."

Tu stack: observe_stream + task_ledger registran tool invocations, pero no por server y sin hash de output. Acción: per-server receipts en ~/.claw/mcp-receipts/{server}.jsonl con timestamp, tool, params resumidos, sha256(output).

3.8 DoS / fatigue (NSA p. 7)

Tu stack: rate-limit es responsabilidad de claude.ai. Riesgo bajo localmente.

5. Action plan re-priorizado

5.1 — Output filtering pipeline (NSA recommendation 6: "Filter and monitor output pipelines and chained execution") P0 — leverage máximo

Implementar el wrapper untrusted_context source="mcp_tool_result" alrededor del output de TODO MCP tool que devuelva contenido externo. El paper NSA es explícito: "Each output must be treated as untrusted input to the next phase of the pipeline."

Tools a envolver (orden por superficie):

  • Figma: get_design_context, get_screenshot, get_figjam, get_metadata, search_design_system
  • Drive: read_file_content, download_file_content, get_file_metadata, search_files, list_recent_files
  • Notion: notion-fetch, notion-search, notion-get-comments
  • Linear: list_comments, get_diff_threads, get_document, get_issue, list_documents
  • Gmail: get_thread, search_threads, list_drafts
  • Calendar: get_event, list_events
  • Mermaid Chart: read_mermaid_file, get_diagram_summary, get_pull_comments, get_issue_comments
  • Vercel: get_runtime_logs, get_deployment_build_logs (logs pueden contener output de terceros)

Estimación: 2-3h en claw_v2/ incluyendo tests. Esto cierra el vector dominante del paper.

5.2 — Gates Tier 3 explícitos (NSA recommendation 2 + 4: "Design for boundaries" + "Constrain and sandbox") P0

Agregar a claw_v2/approvals.py:

  • mcp__claude_ai_Vercel__deploy_to_vercel → Tier 3 duro (deploy externo irreversible)
  • mcp__claude_ai_Vercel__check_domain_availability_and_price → Tier 3 (costo potencial)
  • mcp__claude_ai_Notion__notion-create-pages, notion-update-page, notion-create-database → Tier 2.5 + workspace filter (sólo IDs Pachano)
  • mcp__claude_ai_Linear__save_* para teams no-Pachano → Tier 3
  • mcp__claude_ai_Mermaid_Chart__push_file + create_pr + create_issue → Tier 3 + repo allowlist
  • mcp__claude_ai_Figma__create_new_file + use_figma (writes) + upload_assets → Tier 2.5
  • mcp__claude_ai_Gmail__create_draft → Tier 2.5; cualquier futuro send → Tier 3 duro
  • mcp__claude_ai_Google_Drive__create_file + copy_file → Tier 2.5 + folder allowlist

5.3 — Per-server receipts con hash de output (NSA recommendation 7: "Instrument for logging") P1

Crear ~/.claw/mcp-receipts/{server}.jsonl append-only. Por invocación: timestamp, tool, params resumidos (sin secretos), sha256(output[:8KB]), session_id, decision (allowed/denied/escalated). Conserva 90 días. Hace el patrón Yessenov real y soporta el "Track and patch" del paper (recomendación 8).

5.4 — Capability-change detection (NSA: poor approval workflows) P1

Al iniciar sesión MCP, hash el set ordenado de tools expuestos por cada server. Persistir en ~/.claw/mcp-tool-hashes.json. Si cambia entre conexiones, abrir Tier 3 approval ("Vercel agregó create_billing_invoice — ¿autorizar?"). Cubre el WhatsApp-MCP pattern aplicado a updates silenciosos.

5.5 — Bloqueo preventivo de los 3 OAuth dormidos P2

Daloopa, HubSpot, Windsor.ai: documentar en MEMORY que antes de completar el OAuth flow hay que definir gates específicos para los tools que cada uno expondrá. Sin esto, el flow OAuth completo crearía instantáneamente 3 superficies nuevas sin protección.

5.6 — Diferir HeyGen MCP install P2

Mientras 5.1 + 5.2 no estén durables, mantener pipeline HeyGen vía subprocess CLI. Cuando los controles estén implementados, revisar HeyGen MCP server con due diligence NSA recommendation 1 ("Choose supported MCP projects when possible").

5.7 — Sin necesidad inmediata P3

  • Sandboxing OS-level (AppArmor/seccomp/SELinux — recommendation 4) — no aplica al daemon Mac; los MCP servers viven en infraestructura claude.ai, no en tu host.
  • Filtering outgoing proxy (Squid/tinyproxy — recommendation 2) — el tráfico MCP sale del claude.ai gateway, no del daemon. Tu Mac no tiene visibilidad para filtrar.
  • MCP message signing (recommendation 5) — transport entre claude.ai y daemon usa TLS; tu surface no permite firmar payloads inter-server.
  • Network scan (recommendation 9) — no corres MCP servers locales hoy. Revisitarlo si en el futuro hosteás MCP self-hosted.

6. Validación cruzada — qué del paper NO aplica

El paper está escrito para organizaciones (empresas, agencias gov) hosteando MCP servers self-managed. Tu setup es distinto: sos cliente single-user de servers MCP gestionados por claude.ai. Eso simplifica:

  • ACE / CWE-77, 78, 94, 95: ninguno de los 12 servers ejecuta código arbitrario directo en tu host. Riesgo crítico CWE NO aplica hoy.
  • RBAC at instantiation: el server no se instancia desde tu side; viene pre-configurado por claude.ai.
  • Open MCP server scanning en LAN: no hosteás MCPs.

7. Limitaciones honestas de este audit v2

  • claw_v2/approvals.py NO fue leído en este turno. Las recomendaciones 5.2 asumen el estado documentado; antes de implementar, validar el archivo y comparar con la lista propuesta.
  • Las recomendaciones técnicas son arquitecturales, no implementación lista. Cada item requiere ~30 min de coding + tests para aterrizar.
  • El inventario refleja servers cargados ahora. Un install futuro agrega superficie y debe pasar por este mismo proceso.
  • El paper no menciona explícitamente Anthropic claude.ai bridge — el análisis del bridge es inferencia mía contrastando spec con tool exposure.
