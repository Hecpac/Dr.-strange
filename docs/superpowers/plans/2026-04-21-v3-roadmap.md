## Why v3

v2 grew into a capable prototype with the right primitives (multi-lane LLM
routing, A2A, coordinator/kairos, GraphRAG memory, OOD detection, sub-agents,
NotebookLM, Computer Use). It also grew two god-objects (`brain.py` 1,246 LOC,
`bot.py` 1,026 LOC) and lost durability guarantees: a single Mac crash loses
in-flight pipeline/coordinator work. v3 is **not new features** — it is
**stronger contracts, observability, durability, and portable deployment**.

**One-line thesis:** convert v2 from a prototype-on-a-Mac into a measurable,
durable, portable personal platform.

## Hard-Numbers Audit (baseline, 2026-04-21)

| Module | LOC | Verdict |
|--------|-----|---------|
| `brain.py` | 1,246 | **God orchestrator.** 5 concerns mixed (prompts, voting, gating, snapshots, logging). |

| `agents.py` | 1,346 | Heavy but cohesive. |
| `bot.py` | 1,026 | **God dispatcher.** 10 handlers wired by lambdas. |
| `main.py` | 980 | Composition root getting fat. |
| `telegram.py` | 719 | Transport adapter. Acceptable. |
| Core total | ~17k | 78 files |
| Test total | ~22k | 54 files |

**Own PRD rule violated:** PRD v2.1.6 mandates "250 LOC per file, hard
constraint." Five files break this. This is the structural debt to clear.

**Test gaps:** no e2e flows, no A2A integration tests, no chaos/load tests.

## Execution Order (8 PRs, ~3-4 months)

Order is deliberate: **clean the base before building the tower**. Do not
parallelize. Each PR is small enough to review in one sitting (<800 LOC diff).

### PR#0 — Safety net (1 week)

**Goal:** Snapshot tests of current `brain.handle_message` behavior so PR#1
can refactor without regression.

**Files:**
- Create: `tests/test_brain_snapshot.py`
- Create: `tests/fixtures/brain_cases/` (15 real cases exported from
  `task_outcomes` table)
- Create: `scripts/export_brain_cases.py` (one-shot exporter)

**Steps:**
- [ ] Export 15 representative `task_outcomes` rows (mix of success/failure,
      coding/research/ops) with their `messages` history.
- [ ] Build `BrainTestHarness` that stubs the LLM router with recorded
      responses (replay mode).
- [ ] For each case: assert response text, tool calls emitted, observe events,
      memory writes, approval requests. Snapshot to JSON.
- [ ] Run `pytest tests/test_brain_snapshot.py -v` — must be green before
      starting PR#1.

**Acceptance:**
- [ ] 15+ snapshot tests, all passing on current `main`.
- [ ] Tests are deterministic (no LLM calls, no clock dependency, no FS leaks).

**Risk:** Low. Pure additive.

### PR#1 — Brain decomposition (2 weeks)

**Goal:** Cut `brain.py` from 1,246 LOC to ~300 LOC by extracting 4 services.

**Files:**
- Create: `claw_v2/brain_services/verifier_voting.py` (~250 LOC)
- Create: `claw_v2/brain_services/execution_gating.py` (~200 LOC)
- Create: `claw_v2/brain_services/snapshot_orchestrator.py` (~150 LOC)
- Create: `claw_v2/brain_services/prompt_builder.py` (~250 LOC)
- Modify: `claw_v2/brain.py` (1246 → ~300, becomes stateless wrapper)
- Modify: `claw_v2/main.py` (wire new services)
- Create: `tests/test_verifier_voting.py`, `test_execution_gating.py`,
  `test_snapshot_orchestrator.py`, `test_prompt_builder.py`

**Steps:**
- [ ] Extract `VerifierVotingService`: the 4-lane voting logic + evidence
      aggregation. Pure, no IO.
- [ ] Extract `ExecutionGatingService`: tier-based approval routing,
      autonomy contract checks, capability gating.
- [ ] Extract `SnapshotOrchestrator`: checkpoint pre-staging, rollback
      coordination with `CheckpointService`.
- [ ] Extract `PromptBuilder`: SOUL/USER/HEARTBEAT/profile/lessons/wiki
      composition. Stateless except for cache.
- [ ] `BrainService` becomes: `handle_message`, `execute_critical_action`,
      orchestrates the 4 services. No business rules.
- [ ] Run PR#0 snapshot tests — must remain green.

**Acceptance:**
- [ ] `wc -l claw_v2/brain.py` ≤ 350.
- [ ] PR#0 snapshots still green (no behavioral change).
- [ ] Each new service has its own focused tests.

**Risk:** Medium. Mitigated by PR#0. **Do not merge without snapshot green.**

### PR#2 — HandlerRegistry + bus migration (2 weeks)

**Goal:** Cut `bot.py` from 1,026 LOC to ~400 LOC. Replace lambda-wiring
between handlers with bus events.

**Files:**
- Create: `claw_v2/bot/registry.py` (~150 LOC) — `HandlerRegistry`,
  `Handler` protocol with `can_handle(ctx) -> bool` + `execute(ctx) -> Response`
- Modify: `claw_v2/bus.py` — add typed events `HandlerInvoked`, `HandlerCompleted`
- Modify: 3 pilot handlers to implement the new protocol:
  `browse_handler.py`, `task_handler.py`, `agent_handler.py`
- Modify: `claw_v2/bot.py` (1026 → ~400) — pure dispatcher
- Create: `tests/test_handler_registry.py`

**Steps:**
- [ ] Define `Handler` Protocol + `HandlerRegistry.dispatch(ctx) -> Response`.
- [ ] Migrate 3 pilot handlers; keep the rest on the old API (backward-compat
      via adapter).
- [ ] Replace cross-handler lambdas with `bus.emit("event_name", payload)`.
- [ ] `BotService.handle_text` becomes registry dispatch + bus subscription.
- [ ] Add integration test: send a `/browse` message, assert the registry
      picked the right handler and the bus fired expected events.

**Acceptance:**
- [ ] `wc -l claw_v2/bot.py` ≤ 450.
- [ ] All existing Telegram commands still work (regression: run
      `tests/test_bot.py`).
- [ ] No direct handler-to-handler lambda calls remain in pilot 3.

**Risk:** Low. Old handlers keep working through adapter.

### PR#3 — Typed artifacts (1 week)

**Goal:** Replace ad-hoc dict payloads with typed lineage-aware artifacts.
Now safe because brain.py is decomposed.

**Files:**

  `trace_id`, `created_at`.
- Modify: `claw_v2/types.py` — re-export

- Modify: `claw_v2/brain_services/*` — emit artifacts
- Modify: `claw_v2/pipeline.py`, `coordinator.py`, `kairos.py` — emit artifacts
- Migration: add `artifact_id`, `parent_artifact_id` columns to `observe`
  table (defensive, `IF NOT EXISTS`)
- Create: `tests/test_artifacts.py`

**Acceptance:**
- [ ] Any single Telegram request can be reconstructed by walking
      `artifact_id` lineage backward.
- [ ] `observe` events for brain, pipeline, coordinator all carry the same
      envelope shape.

**Risk:** Low.

### PR#4 — Durable JobService (2 weeks)

**Goal:** Long tasks survive process restart.

**Files:**
- Create: `claw_v2/jobs.py` (~400 LOC) — `JobService`, states `queued |
  running | waiting_approval | retrying | completed | failed | cancelled`
- Modify: `claw_v2/pipeline.py` — pipeline work runs as jobs
- Modify: `claw_v2/notebooklm.py` — NLM podcast/research as jobs
- Modify: `claw_v2/coordinator.py` — coordinator phases checkpoint per step
- Modify: `claw_v2/bot.py` — `/jobs`, `/job_status <id>`, `/job_cancel <id>`
- Modify: `claw_v2/lifecycle.py` — resume incomplete jobs on boot
- Migration: `jobs` table in SQLite (id, type, state, payload_json,
  checkpoint_json, retries, created_at, updated_at)
- Create: `tests/test_jobs.py`

**Acceptance:**
- [ ] Kill `claw` mid-pipeline; relaunch; pipeline resumes from last
      checkpoint without duplicating side effects.
- [ ] `/jobs` lists active jobs with state and elapsed time.

**Risk:** Medium. **Idempotency** is the hard part — every side-effecting
step must be guarded by a dedup key.

### PR#7 — Capability registry + Eval harness (3 weeks)

**Goal:** Adding a new sub-agent is one definition file, zero code in other

- Create: `evals/agent_tasks.jsonl` — 25+ real tasks
- Create: `claw_v2/eval_graders.py` — factuality, cost, latency, risk
  compliance graders
- Create: `scripts/run_agent_evals.py`
- Modify: `.github/workflows/` (if CI exists) — eval gate on prompt changes
- Create: `tests/test_capability_dispatch.py`, `tests/test_evals.py`

**Acceptance:**
- [ ] A SOUL.md change that worsens task success rate by >5% blocks merge.
- [ ] Adding a new specialist agent requires only `claw_v2/agents/<name>/`.

**Risk:** Medium. Eval calibration takes iteration.

### PR#8 — Claw-Core ↔ Claw-Edge spike (2 weeks, time-boxed)

**Goal:** Validate hybrid deploy. Core runs on VPS/Pi 24/7; Edge runs on
Mac for Computer Use only. Communicate via A2A over HTTPS.

**Files:**
- Modify: `claw_v2/a2a.py` — add HTTP transport (currently file-based inbox)
- Create: `claw_v2/a2a_http.py` (~200 LOC) — FastAPI server + client
- Create: `ops/deploy-core.md` — VPS provisioning runbook
- Create: `ops/deploy-edge.md` — Mac edge-only config
- Modify: `claw_v2/config.py` — `CLAW_ROLE=core|edge|monolith` flag

**Acceptance:**
- [ ] Core on a $5 Hetzner VPS responds to Telegram while Mac is asleep.
- [ ] A Computer Use request from Telegram routes Core → Edge over A2A,
      executes on Mac, returns result.

**Risk:** High. This is a spike. If A2A HTTPS adds too much complexity,
park it and keep monolith. Do not productionize without 2 weeks of
observation.

## Success Criteria for v3

Mark complete when ALL true:

- [ ] `wc -l claw_v2/brain.py` ≤ 350
- [ ] `wc -l claw_v2/bot.py` ≤ 450
- [ ] No file in `claw_v2/` exceeds 500 LOC (relaxed from PRD's 250 for

- [ ] A crash at 3 AM loses zero in-flight jobs (PR#4 verified)
- [ ] "Why did the agent do X on Wednesday?" answered from traces alone
      (PR#3 + PR#5)
- [ ] A SOUL.md regression blocks merge (PR#7)
- [ ] Same code runs on Mac, Linux, Docker (PR#6)
- [ ] Adding an agent = 1 definition, 0 code changes elsewhere (PR#7)
- [ ] E2E test suite green: Telegram → handler → skill → approval → execute
- [ ] Daily liveness report posted reliably for 30 consecutive days

## What NOT to Do During v3

- Do not add new Telegram commands until PR#2 lands (HandlerRegistry).
- Do not add new sub-agents until PR#7 (capability dispatch).
- Do not expand Computer Use autonomy until PR#7 (policy via capabilities).
- Do not swap LLM providers without PR#7 eval gate.
- Do not productionize PR#8 (Core/Edge) without 2 weeks of observation.
- Do not start any PR before PR#0 snapshot tests are green.
- Do not merge PRs out of order. The order is the design.

## Definition of "I am in v3"

Claw is in v3 when:
- It survives restarts without losing work.
- It explains its own behavior from data, not logs.
- It runs identically wherever you host it.
- It improves under regression tests, not vibes.
- It expands by policy, not by accumulating special cases.
