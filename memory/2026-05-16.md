## Telegram Autonomy Bug Context

- Verified from `data/claw.db` chats and `observe_stream`, not from persona docs.
- Pattern: Hector often has to say variants of "correlo tu mismo", "Pegale el prompt", "Dale las instructions", or "Abre la app" because the bot drifts into assistant handoff instead of operator execution.
- Concrete failures:
  - 2026-05-11: `correlo tu mismo` followed several fall-through dispatch decisions, but the bot did execute after Hector pushed.
  - 2026-05-15: `browser_use` timed out after 180s during ChatGPT image automation.
  - 2026-05-16: `Abre la app de codex`, `Dale las instructions`, and `Pegale el prompt` fell through as `telegram_actionable_task_no_match` plus `task_intent=disabled_by_flag`; the bot then generated a manual handoff (`Click`, `Cmd+V`, `Enter`) and claimed clipboard success instead of verified paste.
- Root behavioral bug: not "the bot says it is Claude Code", but it behaves like a hosted coding assistant when routers miss. Provider/tool identity must remain internal; Dr. Strange must execute, create a durable task, request approval, block explicitly, or ask a minimal clarification.
- 2026-05-16 fixes added: Telegram imperative router, Computer Use diagnostics, identity/capability binding, and operator handoff guard (`operator_handoff_guard_triggered`) to block manual-final-step responses for actionable requests.
- Later 2026-05-16 hardening: explicit Telegram `ui.open_app`, `ui.inspect_app`, `ui.paste_text`, `ui.set_instructions`, and `ui.submit_prompt` routes now bridge into `ComputerHandler` when local UI capability is available. Paste instructions explicitly prohibit submit/run; submit routes through the stronger Computer Use approval path.
- 2026-05-16 chat audit: recent Telegram messages were not leaking system prompts; the latest system-like replies were expected `Status` / `Tareas pendiente` handlers. Real noise source was Kairos emitting repeated approval-backlog notifications (`kairos_notify_user`) with the same IDs throughout the day. Added approval-backlog fingerprint/cooldown suppression so repeated pending-approval reminders become `kairos_notify_suppressed` unless the approval ID set changes or the alert is critical.
- 2026-05-16 follow-up bug: Hector said `Limpia` after the bot offered to clean duplicate approvals. The message fell through (`telegram_imperative_no_match`, `telegram_actionable_task_no_match`, `task_intent=disabled_by_flag`), then brain said `Arrancando` without a tool/task and later admitted it did not execute. Fix: `Limpia` now routes to `approvals.cleanup_stale_duplicates`, archives stale/duplicate pending approvals through `ApprovalManager.archive`, records a task/result, and `Limpiaste?` reports the recorded cleanup status instead of going to brain/wiki. Added evidence gate to block future `Arrancando`/start claims without task/tool/evidence.
- 2026-05-16 lifecycle fix: root reason Hector has to ask "terminaste?" is that only `autonomous_task_completed/failed` was forwarded to Telegram; task-ledger terminal records from brain tool-use, Telegram imperative handlers, NLM, or stale-task reconciliation could close silently. Added `task_ledger_terminal` Telegram notification path with dedupe against autonomous completion events, plus per-task terminal events when daemon marks stale running tasks as `lost`.
- 2026-05-16 lifecycle watchdog: added a 5-minute `task_lifecycle_watchdog` scheduler job that resumes already-authorized interrupted autonomous coordinator tasks and reconciles false-success ledger rows. This is intentionally conservative: it does not make idle telemetry execute arbitrary pending work, but it prevents authorized tasks from staying silent after interruption.
- 2026-05-16 idle executor promotion path: `CLAW_IDLE_EXECUTOR_ENABLED=1` now performs controlled advancement instead of telemetry-only: starts safe pending session queue/pending-action tasks through `TaskHandler.start_autonomous_task`, resumes durable autonomous coordinator ledger tasks through `resume_idle_autonomous_task`, refuses risky/external objectives, refuses non-durable `in_progress` queue items to avoid duplicates, and treats `already_running` as a no-op rather than a stall attempt.
- 2026-05-16 pending-tasks regression: `Tareas pendientes` routed correctly but polluted the user report by treating closed `failed`/`lost` ledger rows as pending blockers and printing raw tool/runtime errors such as file-too-large and runtime-lost messages. Fix: pending report now shows active queued/running work, approvals, session pending action, and only user-actionable blockers; closed internal failures are counted but omitted from the pending report.
- 2026-05-17 pending-tasks follow-up: the report still felt like the same bad system message because it listed duplicate approvals individually and surfaced a stale user-input blocker from 2026-05-14. Tightened semantics again: normal `Tareas pendientes` now shows compact approval counts, details only active approvals, omits duplicate/stale details, and only surfaces blockers updated within the last 24h. Historical blocked/failed ledger rows require explicit history/audit routes.
- 2026-05-17 pending-tasks tone fix: Hector said the handler still felt like a predetermined/system message, not a natural agent. Updated the normal `Tareas pendientes` response to lead with operational status in natural language ("no tengo tareas corriendo..."), then decisions that actually need attention, then duplicate/noise explanation and a concrete cleanup action.
- 2026-05-17 pending-tasks architecture fix: Hector correctly noted that copy changes were ineffective because `_maybe_handle_pending_tasks_query` intercepted before the brain and did not synthesize from memory. Reworked the route so pending-task questions build an evidence pack from session_state, task_ledger, approval audit, filtered memory facts, and recent messages, then ask the brain for a natural response. Local deterministic wording is now only a fallback when brain synthesis is unusable or errors.
