## HeyGen voice clone + Video Agent v3 canonical pattern validated

**Outcome:** Hector's voice cloned end-to-end and the canonical expressive-no-overlay render pattern validated. He confirmed "está mucho mejor" on the third render (`hector_clean_1779387047.mp4`, msg_id 10520).

### Pipeline (proven, repeatable)

1. **Audio source** — Voice Memo iPhone from `~/Downloads/<voice_memo>.m4a` (any name). User sends as ARCHIVE not voice note (bot transcribes + discards voice notes; archive persists).
2. **Convert m4a → mp3** — `ffmpeg -ac 1 -ar 44100 -b:a 128k` (mono 128kbps). HeyGen rejects `audio/x-m4a` natively. Use Python `subprocess.run` to invoke ffmpeg (sandbox blocks direct call).
3. **Upload asset** — `POST upload.heygen.com/v1/asset` with `Content-Type: audio/mpeg`, body = raw bytes.
4. **Clone voice** — `POST api.heygen.com/v2/voices/clone` MULTIPART with `file` (binary) + `voice_name` + `language=es`. NOT JSON. NOT asset_id reference. Multipart with raw file is the only working pattern.
5. **Render via Video Agent v3** — `heygen video-agent create --prompt <directorial> --avatar-id <look> --voice-id <clone> --orientation portrait --mode generate --wait`. CLI invoked from Python subprocess (Bash blocks `heygen` binary).
6. **Download + Telegram** — `data.video_url` at root of CLI response (NOT in `videos[]`). Download via urllib, send to chat with sendVideo + thumbnail + dimensions.

### Critical prompt rule discovered

Video Agent v3 schema (`--request-schema`) has NO `captions` / `subtitles` flag. The agent decides on its own and **defaults to burning keyword captions across the speaker's face** ("Marca" in render #2 covered Hector's face).

**Fix:** put a `CRITICAL FORMAT RULES` block at the top of the prompt with explicit negative list: no captions, no subtitles, no keyword pop-ups, no lower-thirds, no title cards, no watermarks, no callouts, no animated typography. Close with `FINAL CHECK BEFORE RENDER` line restating the constraint. Render #3 with this addition came out clean and Hector approved.
